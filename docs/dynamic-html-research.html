Dynamic HTML / Interactive Artifact

kimi-web Dynamic HTML 最终方案

最终形态是一个受控的 HTML artifact 系统：模型或工具产出的可交互 HTML 不直接混进 Markdown，也不直接同源打开， 而是以 artifact 卡片进入对话，以受限 iframe 预览，以明确的数据契约保存、回放、下载和继续修改。

日期：2026-06-11 仓库：/Users/moonshot/code/kimi-code-web 范围：apps/kimi-web + protocol + daemon

结论

最终方案采用 HTML artifact + sandboxed iframe。HTML 是正式内容类型，有独立 ID、标题、版本、源码、预览入口和下载入口； 前端只负责展示 artifact 外壳和 iframe，daemon 负责保存 artifact、返回 HTML、加安全响应头。

1

不是 Markdown 增强

`Markdown.vue` 继续只渲染文本和代码块；动态 HTML 走单独的 artifact block。

2

不是同源页面

iframe 默认只开 `allow-scripts`，不加 `allow-same-origin`，不让 HTML 拿到 daemon API、cookie 或宿主 DOM。

3

不是浏览器 VM

WebContainers / Sandpack 解决的是“浏览器里跑项目”。kimi-web 有本地 daemon，完整项目预览应走真实 dev server。

最终体验

聊天里的 artifact 卡片

当模型生成一个 HTML 页面、报告、原型或临时工具时，聊天流里出现一张 artifact 卡片。 卡片展示标题、类型、大小、版本和操作按钮：预览、源码、下载、继续修改。

卡片是对话内容的一部分，所以历史会话回放时仍能看到同一版本，而不是重新从 Markdown 里猜一段 HTML。

右侧或弹层预览

点开预览后，HTML 在专门的 iframe 区域运行。用户可以切换预览和源码，也可以下载完整 HTML。 预览区始终显示 artifact 外壳，避免用户误以为 iframe 内页面就是 kimi-web 原生 UI。

iframe 内页面可以有按钮、筛选器、图表和本地状态，但不能直接调用 daemon、访问工作区文件或联网外传数据。

入口 用户看到什么 系统怎么处理
模型直接生成 HTML artifact 聊天流出现 artifact 卡片，可预览、看源码、下载、继续修改。 消息 content 里保存 artifact 引用；源码存入 artifact store。
工具写入 `.html` 文件 工具卡片提供“打开预览”，files 面板也能预览。 把文件快照提升为 artifact，避免直接 iframe 同源下载路由。
用户打开已有 HTML 文件 文件面板显示“预览 / 源码”切换。 读取文件内容后创建临时 artifact preview，仍按同一 sandbox 策略渲染。
真实 dev server 聊天或任务面板出现 localhost 预览卡。 这是 app preview，不混入 HTML artifact；被页面拒绝嵌入时降级为新窗口打开。

系统形态

核心原则

  • HTML artifact 是对话内容，不是 Markdown 的附属渲染。
  • HTML 源码按版本不可变保存，用户继续修改时生成新版本。
  • iframe 是唯一执行边界，宿主和 daemon 不信任 iframe 内任何数据。
  • bridge 只传递少量白名单消息，例如高度、加载错误、用户请求继续修改。

不做的事情

  • 不使用 `dangerouslySetInnerHTML` 或 `v-html` 执行模型 HTML。
  • 不让 iframe 直接访问 `/api/v1`、`fs:download`、cookies 或 localStorage。
  • 不默认开放任意外部 CDN、fetch、表单提交和弹窗。
  • 不把 WebContainers 当作 HTML artifact 的基础设施。

数据契约

消息内容类型

// packages/protocol/src/message.ts
export const artifactContentSchema = z.object({
  type: z.literal('artifact'),
  artifact_id: z.string().min(1),
  artifact_type: z.literal('html'),
  title: z.string().min(1),
  version: z.number().int().positive(),
  mime_type: z.literal('text/html'),
  size: z.number().int().nonnegative(),
  preview_height: z.number().int().positive().optional(),
});

export type ArtifactContent =
  z.infer<typeof artifactContentSchema>;

Web 层类型

// apps/kimi-web/src/api/types.ts
export interface AppArtifactContent {
  type: 'artifact';
  artifactId: string;
  artifactType: 'html';
  title: string;
  version: number;
  mimeType: 'text/html';
  size: number;
  previewHeight?: number;
}
字段 作用 约束
artifact_id 稳定标识一个 artifact。 只用于查 metadata 和版本，不直接暴露文件路径。
version 指向某次不可变源码快照。 继续修改生成新版本，历史消息仍指向旧版本。
artifact_type 区分 HTML、后续可能的 SVG 或 app preview。 本方案只定义 html。
preview_height 给聊天卡片一个稳定高度。 只能作为建议值，iframe 仍要按上限约束。

安全边界

iframe 默认写法

<iframe
  title="HTML artifact preview"
  sandbox="allow-scripts"
  referrerpolicy="no-referrer"
  loading="lazy"
  src="/api/v1/artifacts/art_123/versions/1:preview"
></iframe>

不加 allow-same-origin。同源 iframe 如果同时有 allow-scripts 和 allow-same-origin， 就可以绕开我们希望建立的隔离边界。

daemon 响应头

Content-Security-Policy:
  default-src 'none';
  script-src 'unsafe-inline';
  style-src 'unsafe-inline';
  img-src data: blob:;
  font-src data:;
  connect-src 'none';
  frame-src 'none';
  base-uri 'none';
  form-action 'none'

script-src 'unsafe-inline' 是为了让 artifact 内部交互能运行；风险靠 iframe sandbox 和关闭网络来收住。

风险 后果 最终处理方式
HTML 与应用同源运行 模型 HTML 能 fetch daemon API，读写文件或触发命令。 artifact route 只服务预览内容；iframe 不加 allow-same-origin；不要直接 iframe `fs:download`。
联网外传数据 artifact 把用户输入、对话内容或本地信息发到外部服务。 connect-src 'none'；图片默认只允许 data: 和 blob:。
伪造宿主 UI 用户误以为 iframe 内按钮属于 kimi-web。 artifact 外壳始终显示标题、来源、sandbox 标识；全屏和外链必须走宿主确认。
postMessage 被伪造 恶意 frame 向宿主发请求。 校验 event.source、schema、artifact instance token；宿主只接受白名单消息。
daemon 本地 API 暴露 其他网页可能尝试访问本机 daemon。 daemon API 应补启动 token 和 Origin 校验；这不是 artifact 独有问题，但 artifact 功能会放大风险。

iframe 高度约束

const MIN_PREVIEW_HEIGHT = 220;
const MAX_PREVIEW_HEIGHT = 720;

function clampPreviewHeight(height: number): number {
  return Math.max(
    MIN_PREVIEW_HEIGHT,
    Math.min(MAX_PREVIEW_HEIGHT, Math.trunc(height)),
  );
}

取舍结论

选项 结论 原因
直接 DOM 插入 HTML 不用 样式、脚本、事件和宿主页面混在一起，支持交互后风险不可控。
纯 iframe srcdoc 只作为内部实现细节 适合临时预览，但缺版本、下载、回放、CSP header 和权限边界，不应作为产品数据模型。
artifact route + sandboxed iframe 采用 能统一消息、文件预览、历史回放、安全头、源码下载和后续交互桥。
真实 dev server iframe 单独能力 适合完整应用预览，不和单文件 HTML artifact 混成一个模型。
Sandpack / WebContainers 不作为基础方案 本地 daemon 能跑真实进程，没必要用浏览器里的 Node runtime 作为第一基础设施。

最终实现清单

✓
定义 artifact message content，并让 messages API、WS projector、mappers、event reducer 都能保留它。
✓
新增 artifact store 和 preview/source/download 路由，preview 路由统一加 CSP。
✓
新增 ArtifactBlock.vue 和 ArtifactIframe.vue，聊天和文件预览复用同一套 iframe 组件。
✓
bridge 只支持 ready、resize、error、request_fix；任何执行 agent、写文件、打开外链的动作都回到宿主确认。
✓
daemon API 增加启动 token / Origin 校验，避免本机其他网页直接打 daemon。
